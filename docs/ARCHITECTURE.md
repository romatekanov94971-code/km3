# Описание архитектуры программного комплекса

## Назначение

Система предназначена для оперативного расчета режимов работы энергообъекта, оценки эффективности энергетического оборудования ТЭС и подбора технических решений для повышения эффективности.

## Основные сценарии

1. Пользователь входит в систему.
2. При первом входе меняет первичный пароль.
3. Вводит нагрузку ТЭС, количество блоков, температуру, влажность, скорость и направление ветра.
4. Получает расчетные показатели.
5. Просматривает графики зависимости теплоотводящей способности от температуры.
6. Экспортирует результаты в CSV/PPTX.
7. Администратор создает пользователей, меняет роли, смотрит аудит и политику безопасности.

## Среда функционирования

- Язык: Python 3.
- Клиент: PyQt6.
- Сервер: FastAPI + Uvicorn.
- Хранилище: SQLite.
- Протокол взаимодействия клиента и сервера: HTTP/JSON, при сетевом развертывании HTTPS.
- Порт по умолчанию: 8000.
- Файловые каталоги: `data/`, `logs/`, `exports/`.

## Ограничения и указания по применению

- В локальном учебном режиме API запускается на `127.0.0.1`.
- Для сетевой эксплуатации необходимо использовать HTTPS: `ENERGY_API_SSL_CERTFILE`, `ENERGY_API_SSL_KEYFILE`.
- Модель оптимизации разрежения в конденсаторе является учебной и должна быть уточнена при наличии паспортных характеристик конкретного оборудования.
- Пароли и тела запросов с паролями не пишутся в аудит.
- Для удаленного сбора аудита задается `ENERGY_AUDIT_REMOTE_URL`.


## Сетевые порты и протоколы

| Компонент | Порт / путь | Протокол | Назначение | Ограничения |
|---|---:|---|---|---|
| PyQt6 GUI → FastAPI | 8000 | HTTP/JSON локально; HTTPS/JSON для сети | Запросы авторизации, расчетов, экспорта, аудита | Для сетевого режима обязательно задать TLS-сертификаты. |
| FastAPI → SQLite | `data/energy_system.sqlite3` | SQLite API | Пользователи, история расчетов, аудит | WAL включен; для production можно заменить на PostgreSQL. |
| FastAPI → файловая система | `logs/`, `exports/`, `data/` | файловый ввод/вывод | audit.log, CSV/PPTX, первичный пароль admin | Права доступа к каталогам должны быть ограничены. |
| FastAPI → remote audit collector | `ENERGY_AUDIT_REMOTE_URL` | HTTP(S) POST JSON | Удаленный сбор событий аудита | Отправка асинхронная через очередь; при заполнении очереди событие не блокирует API. |

## Ограничения SessionManager

Текущий `SessionManager` хранит токены в памяти одного процесса. При перезапуске API все сессии сбрасываются. При запуске нескольких workers каждый worker будет иметь собственное хранилище сессий. Для production нужно заменить реализацию на Redis или таблицу БД.


## Изменения после code review v2

- `AuthenticatedUser.is_admin` реализован без небезопасного `UserRole(str(role))`.
- Аудит использует цепочку sink-объектов: SQLite → remote queue → file log.
- `AuditEvent` содержит `remote_queued`, но не содержит устаревшее поле `remote_sent`.
- GUI logout вынесен в отдельный метод и вызывается явно из пункта меню.
- `AuthService` принимает `IUserRepository`, что снижает зависимость от конкретного класса `UserRepository`.

- Реестр sink-классов аудита расширяется через декоратор `@register_audit_sink(order=...)` без изменения `get_audit_sinks()`.


## Зимний и летний режимы работы

Подсистема расчета поддерживает явный сезонный режим:

| Режим | Назначение |
|---|---|
| `auto` | Автоматический выбор: ниже 5 °C применяется зимний режим, иначе летний. |
| `winter` | Принудительное применение зимних поправок. |
| `summer` | Принудительное применение летних поправок. |

Режим задается в GUI через выпадающий список и в API через поле `operation_mode`.
Фактически примененный режим записывается в `CalculationResult.operation_mode`,
выводится в окне результатов и попадает в CSV/PPTX-отчеты.


## Полнота GUI после проверки функциональности

Исправлено расхождение между расчетным ядром/API и клиентским приложением:

- сезонный режим `auto/winter/summer` теперь проходит через GUI → API → CalculationInput → расчетное ядро;
- в GUI добавлено поле `Разрежение в конденсаторе, кПа`;
- окно графиков теперь содержит вкладки: температура, распределение блоков, конденсатор;
- добавлено окно истории расчетов текущего пользователя;
- окно результатов показывает примененный режим, поправочные коэффициенты, оптимальное число блоков и оптимальное разрежение;
- `CalculationRequest.model_dump()` сохраняет `operation_mode` как строку, поэтому история корректно пишется в SQLite.


## Расчетный след графиков

Графики строятся не из произвольных или заранее заданных чисел. Каждая точка графика содержит `calculation_trace`:

- исходные параметры точки;
- примененный сезонный режим;
- формулы;
- подстановку чисел в формулы;
- расчетные поправочные коэффициенты;
- итоговые значения расчетного ядра.

В GUI это видно в окне «Графики и оптимизация»: справа расположена таблица точек графика, а при выборе строки ниже показывается блок «Расчетный след выбранной точки».


## Визуальное усиление аудита графиков

Чтобы изменение было видно сразу в GUI:

- кнопка в главном окне называется `Графики + аудит расчета`;
- окно графиков называется `Графики + аудит расчета точек`;
- сверху добавлен баннер `АУДИТ ГРАФИКОВ ВКЛЮЧЕН`;
- добавлена отдельная вкладка `Аудит графиков`;
- для каждой точки показывается наличие `calculation_trace`;
- расчет выбранной точки можно скопировать кнопкой `Скопировать расчет выбранной точки`.


## Сводный график во вкладке аудита

Во вкладку `Аудит графиков` добавлен сводный график расчетных точек. Теперь на одной вкладке видно:

- сам график;
- таблицу точек, из которых он построен;
- расчетный след выбранной точки: входные данные, формулы, подстановку и коэффициенты.
