# Архитектура Energy System

## Назначение

Energy System — клиент-серверное приложение для расчёта режимов работы и эффективности энергооборудования ТЭС.

Система решает задачи:

- расчёт КПД энергоблока и ТЭС;
- учёт температуры, влажности, скорости и направления ветра;
- выбор зимнего, летнего или автоматического режима;
- оптимизация разрежения в конденсаторе;
- построение графиков и объяснение расчёта каждой точки;
- хранение истории расчётов;
- аудит действий пользователей и администраторов;
- экспорт результатов в CSV и PPTX.

## Основные подсистемы

| Подсистема | Назначение |
|---|---|
| `app/client` | GUI на PyQt6 |
| `app/server` | API на FastAPI |
| `app/auth` | Аутентификация, роли, сессии, парольная политика |
| `app/calculation` | Расчётное ядро, оптимизация, сезонные режимы, расчётный след графиков |
| `app/storage` | SQLite, репозитории, миграция БД |
| `app/audit` | Журнал аудита, ротация, удалённая очередь |
| `app/reporting` | Экспорт CSV/PPTX и отчётные материалы |
| `app/common` | Общие схемы, исключения и утилиты |

## Клиентская часть

GUI реализован на PyQt6.

Основные окна:

- `LoginWindow` — вход в систему;
- `MainWindow` — ввод параметров и запуск расчёта;
- `ResultsWindow` — результаты расчёта;
- `ChartsWindow` — графики и аудит расчётных точек;
- `HistoryWindow` — история расчётов;
- `AuditWindow` — журнал аудита администратора;
- административные окна создания пользователей, смены ролей и просмотра политики безопасности.

## Серверная часть

API реализован на FastAPI.

Основные группы маршрутов:

- `/auth/*` — вход, выход, смена пароля, пользователи и роли;
- `/calc/*` — расчёт, история, экспорт, аудит.

По умолчанию API работает на:

```text
http://127.0.0.1:8000
```

Для сетевого развёртывания можно использовать HTTPS через переменные окружения с TLS-сертификатами.

## Расчётное ядро

Расчётное ядро принимает `CalculationInput` и формирует `FullCalculationResult`.

Учитываются:

- общая нагрузка ТЭС;
- количество блоков;
- номинальная мощность и КПД блока;
- температура наружного воздуха;
- влажность;
- скорость и направление ветра;
- сезонный режим;
- коэффициент собственных нужд;
- разрежение в конденсаторе.

Выходные показатели:

- нагрузка на энергоблок;
- КПД блока;
- КПД ТЭС брутто;
- собственные нужды;
- КПД ТЭС нетто;
- удельный расход топлива;
- оптимальное количество блоков;
- оптимальное разрежение в конденсаторе.

## Сезонные режимы

Поддерживаются режимы:

```text
auto
winter
summer
```

В режиме `auto` сезон определяется по температуре. В режимах `winter` и `summer` пользователь принудительно задаёт сезонную модель.

## Расчётный след графиков

Каждая точка графика содержит `calculation_trace`.

В нём хранится:

- исходные данные точки;
- применённый режим;
- формулы;
- подстановка чисел;
- поправочные коэффициенты;
- итоговые значения.

Это позволяет объяснить, как было рассчитано каждое значение на графике.

## Аутентификация и роли

Роли:

- `user`;
- `admin`.

Пользователь может выполнять расчёты, смотреть результаты, графики, историю и экспортировать отчёты.

Администратор дополнительно может создавать пользователей, менять роли, смотреть аудит и параметры политики безопасности.

## Аудит

Аудит фиксирует:

- вход и выход пользователей;
- неудачные попытки входа;
- расчёты;
- экспорт;
- административные действия;
- API-запросы.

События записываются:

- в SQLite;
- в файловый журнал;
- при настройке — в удалённый сборщик аудита через очередь.

## Хранение данных

Используется SQLite с WAL-режимом.

Хранятся:

- пользователи;
- история расчётов;
- события аудита.

Файлы экспорта сохраняются в папку:

```text
exports/
```

## Ограничения

- In-memory SessionManager подходит для учебного локального запуска.
- При перезапуске API сессии сбрасываются.
- Для production-развёртывания лучше заменить хранилище сессий на Redis или БД.
- SQLite подходит для учебного проекта; при высокой нагрузке можно заменить на PostgreSQL.
